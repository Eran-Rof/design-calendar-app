# 36. Third-Party Logistics — 3PL (P21 / M13)

**Where:** Tangerine → **Inventory → 🚚 3PL** (`/tangerine?m=three_pl`)

## What it is

A **3PL** is an external warehouse / fulfillment partner that holds your
inventory and ships on your behalf — a *contract* 3PL, distinct from the
marketplace-owned **FBA / WFS** that the Marketplaces module (P12) already
handles. This module is the **provider master** plus **inbound / outbound
shipment tracking** so you know what you've sent to (and received back from)
each 3PL.

Two tabs:

## 36.1 Providers

Register each 3PL you use: **name**, **kind** (Contract 3PL /
FBA / WFS / Other), **contact**, **account #**, and **billing / fee notes** (the
storage + pick-pack terms, free text). The **code** is **auto-generated,
read-only** (`TPL-NNNNN`) — assigned on save, you never type it (any
pre-existing provider codes are kept). A provider can be linked to an
`inventory_locations` row (kind `3pl`) — the location where its stock lives —
and toggled active / inactive.

**Edit a provider (contacts, notes & history).** Click a provider row to open its
editor:
- **Multiple contacts (up to 8)** — each with **name, title, department, email,
  phone** (the same `ContactList` editor as Customer Master, backed by a new
  `tpl_providers.contacts` jsonb column). The single legacy "Contact" still shows
  on the list; the Contacts column shows how many are on file.
- **Notes** — a free-text notes field (plus the existing billing / fee notes).
- **Audit trail** — the same change-history wiring as Customer Master: every edit
  is captured by the T11 audit trigger and shown in the modal's **Audit trail**
  timeline (`GET /api/internal/audit/row-history`).

## 36.2 Shipments

Track stock moving to and from a 3PL:

- **Direction** — *Inbound* (you → 3PL), *Outbound* (3PL → customer/you), or *Return* (3PL → back to you).
- Add a line per item (SKU + qty), a **reference / ASN**, and **carrier + tracking**.
- **Lifecycle:** `draft → in_transit → received → closed` (Send assigns the number `TPL-YYYY-NNNNN`; cancel before it's received).

## 36.3 Waving a sales order to a 3PL (EDI 940 / 945)

When a 3PL holds your stock, you don't ship the sales order yourself — you
**wave** it to the warehouse and they pick, pack, and ship. Tangerine does this
over EDI:

**The round-trip:**

1. **Wave** — once a sales order is **allocated** (or **fulfilling**), release it
   to a 3PL. This creates an **outbound** 3PL shipment (`TPL-YYYY-NNNNN`, status
   **released**) from the SO's allocated lines, and generates an **EDI 940
   (Warehouse Shipping Order)** — the X12 message that tells the warehouse what
   to ship, to whom (the customer ship-to), and from where (the 3PL). The sales
   order is stamped *waved* and moves to *fulfilling*.
2. **Transmit** — the 940 is sent to the 3PL over their configured transport
   (SFTP / AS2 / VAN). **Until the 3PL's real connection is configured** (see
   below), the 940 is **generated and queued** but not transmitted — you can
   still see and re-send it.
3. **945 back** — the 3PL returns an **EDI 945 (Warehouse Shipping Advice)**
   confirming what shipped plus carrier + tracking. Receiving the 945 marks the
   3PL shipment **shipped** and records the tracking number.

**API:** `POST /api/internal/sales-orders/:id/wave` with
`{ tpl_provider_id }`. The 945 return path is
`POST /api/internal/edi/tpl/:provider_id/inbound` (raw X12 945, or a parsed test
body).

> **Inbound goods receipts (944).** The 940/945 round-trip above is for
> *shipping out*. When a 3PL **receives stock into** the warehouse against one of
> your native POs, it sends back an **EDI 944 (Stock Transfer Receipt Advice)**.
> Tangerine turns that into a **draft goods receipt** you confirm and post in
> **Receiving** (which books FIFO + GR/IR). See
> **[Chapter 37 — EDI → 3PL goods-receipt advice (944)](37-edi.md#3pl-goods-receipt-advice-944--draft-po-receipt)**.

### Connecting a 3PL for live EDI — what the operator must provide

Live 940 transmission needs the **3PL's real EDI endpoint + credentials** on the
provider row (Providers tab → the provider's EDI fields), or via env:

- **`edi_protocol`** — `SFTP`, `AS2`, or `VAN`.
- **`edi_endpoint`** — for SFTP: `host[:port]/remote/dir`; for AS2/VAN: the URL.
- **`edi_username`** — the SFTP / partner username.
- **`edi_credential_ref`** — the **name of an environment variable** holding the
  secret (SFTP private key or password / AS2 cert). The secret value itself is
  never stored in the database.

Until these are set, every wave still generates a valid 940 and queues it; the
response says **"generated and QUEUED — not yet transmitted"** with the reason.
SFTP delivery additionally needs the `ssh2-sftp-client` package installed.

## Current scope & what's deferred

This is the **operational tracking** layer. Two things are intentionally **not
wired yet**:

- **Inventory relocation** — receiving a 3PL shipment does **not** yet move FIFO
  layers to the 3PL location. FIFO layers aren't location-scoped beyond their
  receipt today (see the multi-warehouse note, M52); 3PL on-hand will become
  real once that lands.
- **3PL fees** — storage / pick-pack / fulfillment fees aren't auto-posted. The
  Chart of Accounts has the expense accounts for them (e.g. *3 PL Fulfillment*,
  *Logistics Warehouse Expense*, *Storage Fees*); fee posting is a follow-up.

So today the value is a clean record of your 3PL providers and every shipment to
/ from them, with status and tracking.

## 36.x 3PL Inventory Recon — daily differences report (#1112)

**Where:** `/tangerine?m=three_pl_recon` · group **Inventory** · icon 📋

Compares the **3PL's reported on-hand** against **Tangerine's on-hand** so you get
a daily 3PL-vs-Tangerine **differences report**.

**Ingest the 3PL's snapshot.** Pick the provider, then paste or upload its on-hand
file and hit **Ingest & reconcile**. Three formats are accepted:
- **EDI 846** (Inventory Inquiry/Advice) — paste the raw X12 envelope. Parsed by
  `parse846` (`api/_lib/edi/builder.js`): `LIN` = item id, `QTY` (qualifier
  33/ON/QA) = quantity. An audit row is stored in `edi_messages`
  (`transaction_set='846'`).
- **CSV** — `sku,qty` rows (header row is fine; extra columns ignored).
- **JSON** via the API — `{ lines:[{sku,qty}] }`.

Each ingest stores a dated **snapshot** (`tpl_inventory_snapshots` + `_lines`),
resolves every `sku_code` to an item, and recomputes the **differences**
(`tpl_inventory_differences`): per SKU it records the 3PL qty, Tangerine on-hand
**at the provider's location**, and Tangerine **total** on-hand. The grid shows
`Variance = 3PL minus Tangerine`; toggle **Compare vs Location / Total**, filter to
**Mismatches only**, and **Export** (Excel/PDF). It also surfaces SKUs Tangerine
holds at the 3PL location that the 3PL did **not** report (3PL = 0).

> **Location vs Total.** Until 945 receiving relocates FIFO layers to the 3PL's
> `inventory_locations` row (deferred), the **Location** bucket is empty, so
> compare vs **Total on-hand** for a meaningful result. The panel warns when a
> provider has no location link.

**Endpoint:** `POST/GET /api/internal/edi/tpl/:provider_id/inventory-advice`.
Set the provider's **location** in the 3PL master to enable location-scoped
comparison.

### Nightly SFTP auto-pull (#1114)

Fully hands-off: a cron (`/api/cron/tpl-inventory-pull`, **02:30 UTC**) pulls each
provider's latest inventory file over **SFTP** and reconciles it automatically.
Configure it in the recon panel under **⚙ Auto-pull (SFTP)** for the selected
provider:
- **SFTP host[:port]** (`edi_endpoint`), **Username** (`edi_username`),
- **Credential env-var name** (`edi_credential_ref`) — that env var holds the
  password or SSH private key; **the secret never lives in the DB**,
- **Inventory directory** (`inventory_sftp_path`) — the remote folder the 3PL
  drops its nightly file in.

Each night the cron lists that directory, takes the **newest** inventory file
(`*.csv/.txt/.edi/.x12/.846`, or a name containing inv/inventory/onhand/846/stock),
parses it (846 or CSV), runs the same reconcile, and stamps `last_inventory_file`
so the same file is never ingested twice. Uses `ssh2-sftp-client`; reconcile logic
is shared with the manual ingest (`api/_lib/tplInventoryRecon.js`). Manual
paste/upload still works alongside the auto-pull.
